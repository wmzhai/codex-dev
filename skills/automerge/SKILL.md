# AutoMerge

`automerge` 是 `autodev` 之后的正式收尾 skill。它只在用户已经确认分支部署结果满足预期后运行，负责把任务分支收口到主干，并把版本号、正式发布和任务归档做完整。

## 第一规则：先用中文交流

- skill 一触发，第一句就用中文和用户交流。
- 整个执行过程中的检查、阻塞、合并结果、版本号和发布总结默认都用中文。
- 只有用户明确要求英文或双语时，才切换语言。

## Preconditions

只有在以下前提满足时才继续：

- 用户已经明确确认当前分支部署结果符合预期。
- 对应任务已经由 `autodev` 或等价流程推进到“已部署待人工确认”。
- 目标分支当前处于可合并状态。

如果这些前提不成立，要明确说明原因并停止，不要擅自进入正式发布。

## Inputs

从用户请求推断目标任务：

- 如果用户指定了任务编号或文件名，如 `T08` 或 `T08-foo.md`，使用该任务。
- 否则使用 `tasks/` 下状态最接近“已部署待人工确认”的最小编号待办任务文件。

## Workflow

1. 定位目标任务文件并读取全文，确认其中已经包含最新的实现、验证、部署信息。
2. 确认当前分支与任务文件一致；如果当前在其他分支，切到该任务对应分支。
3. 明确检查用户确认是否已经存在：
   - 如果对话中没有明确确认，不要猜测，直接停止。
4. 在进入正式发布前，再同步一次任务文件：
   - 记录用户确认时间或确认结论
   - 记录准备进入主干收尾
   - 保持实现、验证、部署事实为最新状态
5. 准备任务归档：
   - 先把任务文件整理到适合归档的最终结构
   - 默认在当前任务分支把它移动到 `tasks/done/`
   - 如果最终版本号、tag、merge commit 只有在合并后才能知道，允许在正式发布完成后对归档文件再做一次最小补充更新
6. 进入正式发布路径：
   - 如果 gstack `$ship`、`$land-and-deploy`、`$document-release` 与当前仓库流程兼容，优先复用
   - 如果不能完整覆盖当前仓库的实际发布路径，就直接按仓库已有脚本、CI 和平台命令完成合并与发布
7. 版本号职责只在这里处理：
   - 需要 bump 版本号、写 release 文档、打 tag 时，在 `automerge` 中完成
   - 不要把这些动作回推给 `autodev`
8. 完成合并与正式部署后，确认：
   - 主干已更新
   - 部署已完成
   - 归档任务文件包含最终需要保留的 merge / release 事实
9. 用中文向用户汇报：
   - 已合并的分支
   - 主干结果
   - 版本号 / tag
   - 正式部署结果
   - 归档任务文件的位置

## Rules

- `automerge` 只在用户明确确认后使用。
- 允许复用现有 release skills，但不要求机械复用；如果现有 skill 缺少关键行为，就由 `automerge` 自己补足步骤。
- 归档任务文件时，尽量把最终 merge / release 事实也补齐进去。
- 如果主干保护、CI、review gate 或发布 gate 不满足，就停止并明确说明缺口。

## Output

默认返回中文总结，不输出大段源码或完整 patch。

结果中至少包含：

- 目标任务
- 目标分支
- 是否已 merge 到 `main/master`
- 版本号或 tag
- 正式部署结果
- 归档后的任务文件位置
