# Memorize

把项目整理成一套 Codex 能快速上手、快速定位、快速更新的记忆体系。

这个 skill 只做一件事：围绕当前仓库的真实状态，建立或刷新 `AGENTS.md` + `memory/`。目标不是写百科，而是让新来的 Codex session 在最短路径内知道先读什么、问题去哪里找、约束从哪里继承、更新该落到哪些文档。

## 核心原则

- 让 `AGENTS.md` 只保留最高优先级的运行规则、共享约束和最短项目事实。
- 让 `memory/` 负责检索和落点，不负责冗长解释。
- 让每个 memory 文档回答一个明确问题：先去哪里、看哪层、改哪里、验证什么。
- 只写已验证的事实，不写推测。
- 所有新增或更新的项目文档默认使用简体中文撰写，包括 `AGENTS.md`、`memory/` 下文档，以及收敛后的 `CLAUDE.md` 跳转语。
- 旧信息失效时直接删或改，不保留过期路线。

## 目标结构

优先把项目整理成下面这类骨架；具体目录可按项目规模增减：

```text
AGENTS.md
memory/
├── README.md
├── core/
│   ├── symptom-routing.md
│   ├── system-map.md
│   ├── invariants.md
│   └── workflows.md
├── <domain-a>/
│   ├── README.md
│   └── ...
└── <domain-b>/
    ├── README.md
    └── ...
```

## 工作模式

### 1. 新项目初始化

当仓库还没有这套体系时，按以下顺序建立：

1. 识别仓库根目录。
2. 扫描现有入口文档与结构信号：`README.md`、`AGENTS.md`、`CLAUDE.md`、`package.json`、`Cargo.toml`、`pyproject.toml`、`docs/`、`src/`、`app/`、`backend/`、`frontend/`、`packages/`、`tools/`。
3. 识别项目的真实分层、主要业务域、调试入口、部署入口、数据入口。
4. 创建 `AGENTS.md`，只写 Codex 必须先遵守的规则和最短事实。
5. 创建 `memory/README.md`，把热路径和冷路径说清楚。
6. 创建 `memory/core/` 下的基础文档，把系统图、路由图、默认动作、稳定约束写进去。
7. 按业务域补充 `memory/<domain>/` 文档，每个文档只覆盖一个问题域。

### 2. 既有项目刷新

当项目已有 `AGENTS.md` 和 `memory/` 时，按最新代码状态增量更新：

1. 先读现有 `AGENTS.md` 和 `memory/README.md`，再看 `memory/core/` 和相关领域文档。
2. 对照当前代码、目录结构和入口文件，找出新增、移动、废弃或失效的路径。
3. 更新 `AGENTS.md` 中的高优先级规则和最短项目事实，保持它短而硬。
4. 更新 `memory/README.md` 作为导航总表。
5. 更新 `memory/core/` 中的系统图、症状路由、约束和默认工作流。
6. 更新受影响的领域文档，删掉过期分支，补上新落点和新验证路径。
7. 如果某个概念已经不再存在，直接移除对应条目，不要留“可能需要”的废话。

### 3. 处理 `CLAUDE.md`

- 如果项目根目录存在 `CLAUDE.md`，先把其中对 Codex 有价值的内容合并进 `AGENTS.md`，按 `AGENTS.md` 的风格去重、收敛和重写。
- 合并后，`CLAUDE.md` 只保留一小段跳转语句，明确提示 Claude 直接阅读 `AGENTS.md`。
- 不要在 `CLAUDE.md` 里继续保留重复规则、长篇说明或独立知识。

## 记忆文档写法

### `AGENTS.md`

- 只写全局优先级最高的规则。
- 只保留 Codex 真正需要在每次新会话里先知道的东西。
- 写明默认顺序、验证基线、常见禁区、关键目录边界。
- 不要把详细背景、长篇架构说明、面向人类的教程塞进来。
- 如果仓库里原本有 `CLAUDE.md`，优先把它的有效内容并入这里，再让 `CLAUDE.md` 退化成跳转语。

### `memory/README.md`

- 说明 `memory/` 的职责。
- 列出热路径和冷路径。
- 告诉 Codex 遇到不同问题时先去哪个子目录。

### `memory/core/symptom-routing.md`

- 按症状到文档的方式写。
- 目标是让新 session 先定位再读源码。
- 一条路由只保留最直接的文件入口。

### `memory/core/system-map.md`

- 只描述仓库的层级边界和数据边界。
- 让 Codex 知道哪个目录负责什么。

### `memory/core/invariants.md`

- 收录稳定且容易误判的约束。
- 例如数据格式、时间规范、命名边界、禁止做法、默认验证基线。

### `memory/core/workflows.md`

- 记录新会话默认动作和修改后的验证基线。
- 只写高频、可复用、对上手速度有帮助的流程。

### 领域文档

- 按项目真实业务域分组。
- 每篇只回答一个明确问题。
- 文档标题要能被快速扫读。
- 每篇都要列出关键入口文件和最可能的修改落点。

## 验证与更新标准

在整理或刷新记忆体系时，优先确认这些信息：

- 项目根目录在哪里。
- 主要执行入口在哪里。
- 共享层和业务层怎么分。
- 调试入口在哪里。
- 部署、恢复、外部联调入口在哪里。
- 哪些规则必须被 Codex 反复记住。
- 哪些文档已经过期，应该删除或重写。

如果项目后来发生了结构变化，优先让 `memory/` 追上真实状态，再考虑补充其它文档。

## 输出要求

完成这个 skill 对应的任务时，最终结果应当让下一个 Codex session 能直接做到：

1. 先读对文件。
2. 快速找到相关代码。
3. 知道该遵守的约束。
4. 知道该更新哪份记忆文档。
5. 所有产出的项目文档默认使用简体中文，避免混入英文默认稿。
