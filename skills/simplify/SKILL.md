# Simplify

把给定 diff 重构成更清晰、更小、但保持行为不变的版本。

把它视为语义不变的清理，而不是功能开发。它通常作为 `checktask` 的内部步骤被调用；在 codev + gstack 流程里，也常用于第一次普通 commit 或 `checkpoint` 前先把当前 patch 收窄；没有 task 时也可以单独使用。

## Inputs

输入通常是代码 diff，或一个明确指向当前 patch 的请求。

如果作为 `checktask` 的内部步骤调用，输入应直接沿用 `checktask` 已确定的相关最小 diff，不要重新扩大范围或再次触发另一轮 `simplify`。

如果用户单独使用且没有直接提供 diff，则先从当前工作区提取最相关、最小范围的 diff，再进行精简。

如果当前目标是为普通 commit、`checkpoint`、`design-review` 或 `qa` 准备 clean tree，则只处理当前尚未提交的那一小块 patch，不要顺手扩大到无关历史改动。

## Goals

- 简化控制流。
- 降低嵌套层级。
- 消除重复逻辑。
- 在局部且不改变语义的前提下改进命名。
- 优先使用周边代码已经采用的惯用写法。
- 在可能时让 patch 更小、更易审查。

## Constraints

- 不要改变行为。
- 不要改变公共 API，除非原始 diff 已经在改，且为了保持原意必须如此。
- 不要引入新依赖。
- 不要无故扩大改动范围；只有在保持结果连贯时，才允许极小的相邻清理。
- 不要静默删除具有运维或语义意义的错误处理、校验、日志或注释。

## Workflow

1. 在编辑前先阅读 diff 及其周边上下文。
2. 找出重复逻辑、不必要的分支、过度具体的命名，以及可以消除的临时状态。
3. 优先采用直接、低风险的变换：
   - 当可读性更好时，内联只使用一次的临时变量
   - 合并冗余条件分支
   - 只有在明显减少重复时，才提取重复的局部表达式
   - 用附近已经存在的惯用模式替换临时拼凑的写法
4. 直接在工作区做最小必要修改，不要为了展示结果而回显大段代码、整段函数或完整 patch。
5. 对每一处改动重新核对原始行为，包括边界情况和错误路径。
6. 仅返回简短结果摘要：改了什么、为什么更简单、是否有无法安全精简的部分。

## Output

默认返回简短文字摘要，不输出完整 unified diff、不粘贴大段源码、不使用代码块。

只有在用户明确要求查看 patch 或具体代码时，才提供必要的最小片段。

## Judgment

宁可不改，也不要做有风险的“简化”。

如果某种简化必须以行为变化为代价，就保留更安全的版本。
