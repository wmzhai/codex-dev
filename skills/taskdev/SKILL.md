# TaskDev

`taskdev` 是 task plan 的手动执行器。它把已经审核通过的 task 文件转成实际代码改动，并把实现过程同步回任务文档。

它只覆盖 task 分支内的“选择任务 -> 校准 plan -> 编码 -> 最小本地验证 -> 同步 task”这一段。它不是 `autodev` 的替代发布链，不负责完整 QA、分支部署、归档或主干收尾。

## 第一规则：先用中文交流

- skill 一触发，第一句就用中文和用户交流。
- 整个执行过程中的进度更新、阻塞说明、实现摘要和结果汇报默认都用中文。
- 只有用户明确要求英文或双语时，才切换语言。

## Inputs

从用户请求推断目标任务：

- 如果用户指定了任务编号或文件名，如 `T08` 或 `T08-foo.md`，使用该任务。
- 否则使用 `tasks/` 下任务号最小的待办任务文件，只识别文件名匹配 `^T\\d+-.*\\.md$` 的任务文件，并忽略 `tasks/done/` 与其他非任务文件。

## End State

完成时应尽量达到以下状态：

- 目标 task 已在对应 task 分支上完成当前 plan 范围内的代码实现。
- 已运行最小必要的本地验证，并把结果写回任务文件。
- 对应 `tasks/Txx-*.md` 已同步实际采用的实现路径、重要代码改动、验证结果和剩余缺口。
- 当前结果已经适合继续进入 `$simplify`、`$checkpoint`、`$design-review`、`$review`、`$qa` 或后续手动验证。

`taskdev` 默认不负责以下动作：

- 不自动做完整 QA、浏览器验证或分支部署
- 不归档到 `tasks/done/`
- 不 merge 到 `main/master`
- 不 bump 版本号、打 tag、创建 PR
- 不默认 commit 或 push；需要稳定中间点时，后续单独使用 `$checkpoint`

## Workflow

1. 定位仓库根目录并选择目标任务文件；如果没有待办任务，明确告知并停止。
2. 基于目标任务文件确定工作分支：
   - 取任务文件去掉 `.md` 后的文件名，作为该任务的工作分支名。
3. 检查当前 git 分支与工作区：
   - 如果当前已在目标任务分支，继续后续步骤。
   - 如果当前不在目标任务分支，且工作区足以安全切换，则切到该任务分支。
   - 如果该分支不存在，就基于当前 `HEAD` 新建并切过去。
   - 如果存在未提交改动导致无法安全切换分支，明确报告阻塞并停止。
4. 完整读取任务文件，提取：
   - 目标与背景
   - 验收标准
   - 依赖任务
   - 约束、备注、相关代码入口
   - 已有的 `Implementation Plan` 与 `Validation Plan`
5. 检查依赖任务是否已完成：
   - 如果前置任务未完成或明显缺前提，不要绕过。
   - 先把阻塞原因写入任务文件，再向用户说明并停止。
6. 初始化或补齐任务文件中的过程记录区：
   - 保留原有目标、范围和验收结构。
   - 保留并持续维护已有的 `Implementation Plan` 与 `Validation Plan`。
   - 如有必要，补充 `## Status`、`## Execution Log`、`## Implementation Notes`、`## Validation Notes`、`## Remaining Gaps` 等小节。
7. 在开始编码前先校准 task plan：
   - 优先完整读取任务文件中已有的 `## Implementation Plan` 与 `## Validation Plan`。
   - 如果代码现状与原计划有轻微漂移，先在任务文件里直接更新这些小节，再开始编码。
   - 如果漂移已经上升到产品范围或架构路径冲突，明确说明并停止；必要时建议用户回到 task 文件继续收敛，或补跑 `$plan-eng-review` / `$autoplan`。
8. 分阶段实现代码：
   - 优先按可验证的小块推进，而不是一次性铺开。
   - 优先复用当前 repo 已有模式，而不是凭空设计新结构。
   - 每完成一个重要里程碑，就更新一次任务文件中的执行记录和实现说明。
9. 做最小必要本地验证：
   - 优先跑与本次改动直接相关的测试、lint、build 或脚本检查。
   - 如果没有现成自动化验证入口，就做最小可解释的静态检查或手动验证，并把缺口写清楚。
   - 不默认触发 `$design-review`、`$review`、`$qa` 或部署；这些属于后续独立门禁。
10. 收尾前再次同步任务文件：
   - 实际实现与原计划的差异
   - 已改动的重要文件与模块
   - 已完成的最小验证
   - 剩余缺口、风险和建议的下一步
   - 当前状态建议写成“实现完成待进一步验证”或语义等价的表述
11. 用中文向用户汇报：
   - 当前 task
   - 当前分支
   - 已完成的代码实现
   - 已跑的最小验证
   - 尚未执行的下游门禁
   - 下一步建议使用 `$simplify`、`$checkpoint`、`$design-review`、`$review`、`$qa` 或继续人工处理

## Rules

- 默认一次只处理一个 task。
- 默认按最小整数任务号选择待办任务，不扫描 `tasks/done/`。
- 任务文件不是只读输入；实际实现路径变化时，必须按事实同步 task。
- 不要为了“顺手收口”而替用户执行 QA、部署、归档或发布。
- 不要假设必须 commit 才算完成；`taskdev` 默认停在代码与任务文档都已更新的实现阶段。
- `autodev` 已经内含这一阶段；如果用户明确要“自动推进到可部署待确认”，直接使用 `autodev`，不要先跑一遍 `taskdev` 再重复执行。

## Hard Blockers

只有在以下情况才停下来问用户：

- 产品范围或行为出现冲突性决策，且无法从 task、repo 或现有模式中安全收敛
- 依赖任务未完成，继续执行会制造错误前提
- 环境或未提交改动导致无法安全切到目标 task 分支
- 最小必要验证暴露出多条高影响实现路径，无法安全默认选择

如果遇到硬阻塞：

- 先把当前状态和阻塞原因写入任务文件
- 再用中文向用户说明已经完成到哪一步、缺什么、需要用户给出什么最小输入

## Output

默认返回中文总结，不输出大段源码或完整 patch。

结果中至少包含：

- 任务文件名
- 当前分支
- 当前阶段状态
- 已完成的重要实现
- 已执行的最小验证
- 剩余风险或待验证项
- 下一步推荐的下游 skill 或手动动作
