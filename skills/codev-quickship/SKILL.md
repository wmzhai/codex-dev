# QuickShip

`codev-quickship` 是面向小改动的轻量收尾 skill。它基于当前本地工作状态直接完成“快速提交并推主干”：如果当前在分支上，就做“并主干 + 推主干”；如果已经在主干上，就做“commit + push”。它不走标准 PR / release 链路，也不调用 `$ship`、`$land-and-deploy`、`$document-release`。

## 第一规则：先用中文交流

- skill 一触发，第一句就用中文和用户交流。
- 整个执行过程中的检查、阻塞、合并结果和推送结果默认都用中文。
- 只有用户明确要求英文或双语时，才切换语言。

## Preconditions

只有在以下前提满足时才继续：

- 用户已经明确确认当前结果符合预期，并接受直接 push 主干。
- 当前仓库允许直接推送主干；如果有 branch protection、权限限制或必须走 PR 的规则，要明确说明并停止。
- 如果当前不在主干，当前分支必须处于可合并状态；如果存在未解决冲突或明显未收敛的工作区，先处理后再继续。

如果这些前提不成立，要明确说明原因并停止，不要擅自直推主干。

## Inputs

从用户请求与当前仓库状态推断目标：

- 默认以当前 git 分支作为工作起点。
- 目标分支优先使用仓库默认主干；如果无法判断，则按 `main` 优先、`master` 兜底。
- 如果当前分支名与某个 `tasks/Txx-*.md` 对应，可把该任务文件作为可选同步对象；如果没有对应 task，就不要为了补 task 而扩大范围。

## Workflow

1. 确认当前分支、目标主干与用户确认结论：
   - 如果对话中没有明确出现“可以直接提交并推主干”的确认，不要猜测，直接停止。
2. 检查当前工作区：
   - 如果有未提交改动，但它们显然属于本次 quickship，可先在当前分支做一次最小提交再继续。
   - 如果改动混杂、来源不清或存在高风险文件，不要替用户猜，直接停止并说明需要先整理工作区。
3. 可选同步 task 文档：
   - 如果当前分支有明确对应的 task 文件，补记“已确认可 quickship”的事实和最小必要的实现/验证结论。
   - 不要因为 quickship 去扩展 repo 级文档同步；没有 task 文件就直接跳过。
4. 根据当前所在分支决定 quickship 路径：
   - 如果当前不在主干：先同步本地主干到最新远端状态，再切到目标主干，把当前分支合并进主干；能 fast-forward 就保持简单线性历史，不能 fast-forward 时再做普通 merge。
   - 如果当前已经在主干：直接在当前主干上整理并提交这次小改动，不额外创建或依赖 feature branch。
5. 直接推送主干：
   - 把本地主干推到远端主干。
   - 如果推送失败、被保护规则拦截或权限不足，明确报告并停止，不要改走其他发布技能。
6. 如有 task 文件且本次已经补记了 quickship 事实，可在最小必要范围内补上最终 commit / merge / push 结果。
7. 用中文向用户汇报：
   - 起始分支
   - 目标主干
   - 走的是“branch merge”还是“main commit + push”
   - 是否已成功完成 commit / merge / push
   - 是否同步了 task 文件
   - 如果失败，卡在哪个 git / 权限步骤

## Rules

- `codev-quickship` 只用于小改动、低流程摩擦的快速收口场景。
- 不要调用 `$ship`、`$land-and-deploy`、`$document-release`。
- 不要创建 PR。
- 不要 bump 版本号、写 CHANGELOG、打 tag。
- 如果主干 push 之后会触发仓库自己的自动部署，这是仓库现状；`codev-quickship` 不负责等待部署或做 canary 验证。
- 如果用户需要正式发布、版本号、tag、repo 级文档同步或标准 PR 流程，应该转去 `codev-automerge` 或 gstack 正常发布链路。

## Output

默认返回中文总结，不输出大段源码或完整 patch。

结果中至少包含：

- 起始分支
- 目标分支
- 本次走的是 branch merge 还是 main commit + push
- 是否已成功完成 commit / merge
- 是否已成功 push 主干
- 是否触碰 task 文件
- 如果未完成，具体阻塞点
