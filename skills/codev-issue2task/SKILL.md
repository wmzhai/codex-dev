# Issue2Task

把 issue 或用户直接描述的需求先理解清楚，再结合真实代码落成可以直接执行的任务计划文件。

这个 skill 同时负责需求梳理、任务拆解和实现方案收敛；但它先产出“需求理解 + 待确认清单”，只有在用户确认所有关键细节后，才会写出 task 文件。输出的 task 文件本身才是后续执行要审核的 plan。

## 第一规则：先用中文交流

- skill 一触发，第一句就用中文和用户交流。
- 整个执行过程中的需求判断、阻塞说明、计划摘要和结果汇报默认都用中文。
- 只有用户明确要求英文或双语时，才切换语言。

## Inputs

从用户请求推断范围：

- 如果用户没有输入任何参数，先在当前仓库的 open issues 里找编号最小的一条，只处理这一条。
- 如果用户指定了单个 issue，如 `42` 或 `#42`，只处理该 issue。
- 如果用户显式输入了多个 issue 编号，如 `42 43 44`、`42,43,44`、`42, 43 44` 或混写 `#42 #43,44`，把它们视为同一组 issue 集合；默认为这组 issue 创建一个总体 task，适合合并多个 issue 做一个版本。
- 如果用户带了 `gh issue list` 可接受的过滤条件，如 `--label`、`--milestone`、`--assignee`，按这些条件取 issue。
- 如果用户在 `$codev-issue2task` 后面直接给了一段自然语言需求，而不是 issue 编号或 `gh issue list` 过滤参数，把这段文字当作直接输入，不查 issue 列表。

## Workflow

1. 确认当前目录是 git 仓库；如果这次请求需要读取 GitHub issue，再额外确认 `gh` 可用。缺少任一必需前提时明确报告阻塞并停止。
2. 读取输入源：
   - 直接文本模式：把用户附带的自然语言需求当作输入源，不查询 issue。
   - 无参数模式：先列出当前仓库 open issues，选编号最小的一条，再用 `gh issue view` 读取。
   - 单个 issue 用 `gh issue view`。
   - 用户显式输入多个 issue 编号时，按逗号或空白分隔解析，去掉可选的 `#` 前缀、去重并按编号升序规范化；然后对每个 issue 用 `gh issue view` 读取，并把这组 issue 作为一个组合输入源继续分析。
   - issue 过滤条件模式用 `gh issue list --state open --json number,title,body,labels,comments,milestone,assignees --limit 200`。
3. 仔细阅读输入源，然后大量阅读相关代码建立上下文：
   - 当前行为是什么。
   - 涉及哪些模块、文件和调用关系。
   - 有哪些输入源没写清但代码里真实存在的约束。
   - 影响范围可能扩展到哪里。
4. 基于输入源和代码现状先自行收敛需求理解，并先向用户输出一轮中文讨论结果，而不是直接写文件：
   - 明确功能边界和 out-of-scope。
   - 补全关键用户场景。
   - 标出边界条件、异常流、权限和空状态。
   - 把验收标准草案写成可验证的表述。
   - 按“问题复述 -> 当前代码现状 -> 关键判断 -> 待确认清单 -> 推荐方案 -> 确认门禁”的固定提纲输出。
   - 只有在歧义会显著改变任务拆分、验收标准或实现路径，且无法从代码或输入源上下文推断时，才中途向用户提一个最小问题。
5. 等待用户把关键细节确认完整；只有当用户明确确认“按当前理解继续”或等价表达后，才进入写文件阶段。
6. 在需求边界稳定后，继续基于真实代码写出可直接实施的 plan：
   - 推荐一条默认实现路径，并简短说明放弃其它路径的原因。
   - 标出计划修改或新增的关键文件、模块、接口和脚本。
   - 写清实施顺序、关键假设、风险点和最小验证链。
   - 优先复用当前 repo 已有模式，而不是凭空设计一套新结构。
7. 任务拆分默认保守处理：
   - 默认一个输入源对应一个任务，不主动过度拆分。
   - 用户显式输入多个 issue 编号时，默认把整组 issue 当作一个版本任务，不因为它们本身是多个 issue 就自动拆成多个 task。
   - 只有在输入源明显过大、无法作为一次可控实现时，才拆成多个可独立落地的任务。
   - 只有在多个小 issue 高度耦合、拆开会制造无意义边界时，才合并为一个任务。
   - 同一个输入源如果拆成多个任务，也按连续的新整数任务号分别创建，不再使用字母后缀。
8. 创建或复用仓库根目录的 `tasks/`。
9. 分配编号并保留已有任务为默认行为：
   - 扫描 `tasks/` 和 `tasks/done/` 中已有任务文件名的基础整数部分，取最大值顺延；如果还没有任务，从 `T01` 开始。
   - 同一个输入源拆成多个任务时，按连续的新整数任务号分别创建，例如 `T12`、`T13`、`T14`。
   - 如果任务来自单个 issue，先在 `tasks/` 和 `tasks/done/` 里查找是否已经有任务通过文件内容映射到同一个 issue 编号；兼容历史任务中的 `## Source Issues` 段和新任务标题下方的 `Issue:` / `Issues:` 行。只有用户明确要求 regenerate 或 overwrite 时才覆盖或重建。
   - 如果任务来自用户显式输入的多个 issue 编号，先用规范化后的完整 issue 集合去查找是否已经有任务映射到完全相同的 issue 集合。不要因为存在单 issue 或部分重叠的旧任务就默认阻止创建；这类重叠在“合并多个 issue 做一个版本 task”的场景下是允许的，但要在结果摘要里提醒用户注意已有重叠任务。
   - 直接文本模式默认按新编号创建任务；只有用户明确要求覆盖现有文件时才覆盖。
10. 只有在用户确认后的这一轮，才为本次将要创建的每个 task 计算目标文件名。
11. 为每个任务分别执行以下动作：
   - 默认不新建分支、不切分支，直接在当前分支写入 task 文件。
   - 必须遵守当前仓库或工作区的分支规则；如果项目约定默认在 `main` 主干开发，就不要为 task 创建分支。
   - 只有用户明确要求为本次 task 建分支时，才按用户要求创建或切换分支，并在结果摘要里说明。
12. 为每个任务写入 `tasks/T{nn}-{slug}.md`，模板如下：

```markdown
# T{nn}: {任务标题}

{若源自单个 issue，紧跟标题写 `Issue: #123`；若源自多个 issue，写规范化后的 `Issues: #123, #124`；若源自直接文本，写 `Source: User prompt`}

## Source Context

- #123 Issue 标题
- 或：用户输入摘要

## Task Description

{整理后的完整需求，反映输入源、代码现状与用户已明确提供的信息所收敛出的范围和目标}

## Acceptance Criteria

- [ ] 标准 1
- [ ] 标准 2

## Related Code

- `path/to/file.ts` - 当前行为、关联模块或约束说明

## Implementation Plan

### Proposed Approach

{推荐实现路径，包含关键数据流、边界处理或复用策略}

### File Changes

- `path/to/file.ts` - 计划修改原因
- `path/to/new-file.ts` - 计划新增原因

### Execution Order

1. 第一步
2. 第二步

### Assumptions / Open Questions

- 假设 1
- 如无则写 `None`

## Validation Plan

- [ ] 验证 1
- [ ] 验证 2

## Dependencies

{任务编号列表或 "None"}

## Notes

{可选：评论中的技术约束、需求细节、取舍说明}
```

13. 输出简短摘要，包含分析了多少 issue 或直接输入源、经过了哪些确认点、创建了多少任务、任务文件路径、当前分支、结果依赖链，以及“用户接下来应审核 task plan，再进入 `$codev-taskdev`”的提示。

## Rules

- 先做需求澄清，再写任务文件；不要把输入源原文直接机械搬运成任务。
- 默认把“先讨论确认，再落 task”视为硬门禁；没有用户明确确认前，不要创建 `tasks/Txx-*.md`。
- 重点保留会影响实现的技术细节、约束和用户已明确提供的信息。
- 任务文件必须同时能服务“需求对齐”和“后续直接执行”，不要把实现计划再外包给另一个 skill。
- 基础任务号始终按 `tasks/` 和 `tasks/done/` 的最大现有编号顺延，而不是复用 issue 编号。
- 显式 issue 编号列表支持逗号、空格或混合分隔，并兼容 `#123` / `123` 混写。
- 用户显式输入多个 issue 编号时，默认产出一个总体 task；只有用户明确要求拆开，或代码分析证明合并后无法安全实施时，才拆成多个任务。
- 同一个输入源如果拆成多个任务，也必须拆成连续的不同整数任务号，不再使用字母后缀。
- 任务排序统一按整数任务号升序；后续 `codev-taskdev` 也按这套顺序选择“最小”任务。
- `tasks/` 不需要额外索引文件；任务文件名本身就是索引，命名规则同时定义了唯一标识和处理顺序。
- 任务状态和映射信息只写在具体任务文件里。
- 如果任务源自 issue，必须在标题下方第一行写出 issue 编号映射，保证后续能从 task 反查到 issue。
- 多 issue 合并场景的去重判断以完整 issue 集合为准，而不是任一单个 issue 命中就停止。
- 不输入参数时，不要扫全量 issue 后批量生成任务；只处理当前 open issues 中编号最小的一条。
- 默认不为 task 创建或切换分支；遵守当前仓库或工作区已有分支规则。只有用户明确要求分支时，才创建或切换分支。
- 默认先完成确认回合，不把“直接写入任务文件”当成起始步骤；没有用户明确确认前，不落盘任务文件。
- task 文件里的 `Implementation Plan` 和 `Validation Plan` 必须足够具体，使用户审核后可以直接开始实现，而不是再补一轮规划。
- 不要提交代码。文件创建就是这个 skill 的最终副作用。

## 标准讨论提纲

当进入确认回合时，优先按下面的固定结构输出，而不是自由发挥：

1. `问题复述`
   - 用 1 到 3 句话概括我对 issue / 需求的理解。
   - 明确这次要解决什么，不解决什么。

2. `当前代码现状`
   - 现有实现在哪里。
   - 哪些部分已经满足，哪些部分还缺口最大。
   - 我认为最相关的模块、文件或调用链是什么。

3. `关键判断`
   - 这次需求的真实范围。
   - 是否需要兼容旧行为。
   - 是否涉及 UI、API、数据迁移、权限或测试链路。
   - 是否更适合一个 task，还是需要拆 task。

4. `待确认清单`
   - 只列会影响实现路径的问题。
   - 每条问题尽量附一个我倾向的默认答案，方便用户直接确认或修正。
   - 如果问题很多，优先只问最阻塞的 3 到 5 个。

5. `推荐方案`
   - 给出默认实现路径。
   - 简短说明为什么选它，以及为什么不选其他路径。
   - 标出主要风险和最小验证方式。

6. `确认门禁`
   - 明确说明在用户确认前不会写 `tasks/Txx-*.md`。
   - 用户只要回复“按这个理解继续”或逐条修正，就可以进入落盘阶段。

## 标准输出模板

```text
我对这个 issue 的理解是：
- ...

我看到的代码现状是：
- ...

这里有几个会影响实现的待确认点：
1. ...
2. ...
3. ...

我的默认方案是：
- ...

如果你确认以上理解，我再把它写成 task 文件。
```
