# Issue2Task

把 issue 或用户直接描述的需求先理解清楚，再落成可以执行的任务文件。

这个 skill 聚焦需求梳理和任务拆解，不负责代码实现方案。

## Inputs

从用户请求推断范围：

- 如果用户没有输入任何参数，先在当前仓库的 open issues 里找编号最小的一条，只处理这一条。
- 如果用户指定了单个 issue，如 `42` 或 `#42`，只处理该 issue。
- 如果用户带了 `gh issue list` 可接受的过滤条件，如 `--label`、`--milestone`、`--assignee`，按这些条件取 issue。
- 如果用户在 `$issue2task` 后面直接给了一段自然语言需求，而不是 issue 编号或 `gh issue list` 过滤参数，把这段文字当作直接输入，不查 issue 列表。

## Workflow

1. 确认当前目录是 git 仓库；如果这次请求需要读取 GitHub issue，再额外确认 `gh` 可用。缺少任一必需前提时明确报告阻塞并停止。
2. 读取输入源：
   - 直接文本模式：把用户附带的自然语言需求当作输入源，不查询 issue。
   - 无参数模式：先列出当前仓库 open issues，选编号最小的一条，再用 `gh issue view` 读取。
   - 单个 issue 用 `gh issue view`。
   - 多个 issue 用 `gh issue list --state open --json number,title,body,labels,comments,milestone,assignees --limit 200`。
3. 仔细阅读输入源，然后大量阅读相关代码建立上下文：
   - 当前行为是什么。
   - 涉及哪些模块、文件和调用关系。
   - 有哪些输入源没写清但代码里真实存在的约束。
   - 影响范围可能扩展到哪里。
4. 基于输入源和代码现状先自行收敛需求理解，并把关键判断写进任务文件：
   - 明确功能边界和 out-of-scope。
   - 补全关键用户场景。
   - 标出边界条件、异常流、权限和空状态。
   - 把验收标准写成可验证的表述。
   - 只有在歧义会显著改变任务拆分或让验收标准失真、且无法从代码或输入源上下文推断时，才中途向用户提一个最小问题。
5. 任务拆分默认保守处理：
   - 默认一个输入源对应一个任务，不主动过度拆分。
   - 只有在输入源明显过大、无法作为一次可控实现时，才拆成多个可独立落地的任务。
   - 只有在多个小 issue 高度耦合、拆开会制造无意义边界时，才合并为一个任务。
   - 同一个输入源如果拆成多个任务，共用同一个新基础任务号，并按 `a` / `b` / `c` 追加后缀。
6. 创建或复用仓库根目录的 `tasks/`。
7. 分配编号并保留已有任务为默认行为：
   - 扫描 `tasks/` 和 `tasks/done/` 中已有任务文件名的基础整数部分，取最大值顺延；如果还没有任务，从 `T01` 开始。
   - 同一个输入源拆成多个任务时，共用这个新基础任务号，并按 `a` / `b` / `c` 追加后缀。
   - 如果任务来自 issue，先在 `tasks/` 和 `tasks/done/` 里查找是否已经有任务通过文件内容映射到同一个 issue 编号；兼容历史任务中的 `## Source Issues` 段和新任务标题下方的 `Issue:` / `Issues:` 行。只有用户明确要求 regenerate 或 overwrite 时才覆盖或重建。
   - 直接文本模式默认按新编号创建任务；只有用户明确要求覆盖现有文件时才覆盖。
8. 在写入任何 task 文件之前，先新建一个分支，并切到这个新分支：
   - 默认分支名使用本次将创建的第一个 task 文件去掉 `.md` 后的文件名，例如 `T12-fix-checkout-empty-state`。
   - 如果同一个输入源拆成多个任务，仍只创建一个新分支，并使用第一个 task 文件的文件名作为分支名；本次生成的全部 task 文件都写在这个新分支里。
   - 如果同名分支已存在，为保证“新建一个 branch”，应改用带短后缀的新分支名，并在结果摘要里说明。
9. 为每个任务写入 `tasks/T{nn}{suffix?}-{slug}.md`，模板如下：

```markdown
# T{nn}{suffix?}: {任务标题}

{若源自 issue，紧跟标题写 `Issue: #123` 或 `Issues: #123, #124`；若源自直接文本，写 `Source: User prompt`}

## Source Context

- #123 Issue 标题
- 或：用户输入摘要

## Task Description

{整理后的完整需求，反映输入源、代码现状与用户已明确提供的信息所收敛出的范围和目标}

## Acceptance Criteria

- [ ] 标准 1
- [ ] 标准 2

## Related Code

- `path/to/file.ts` - 当前行为、关联模块或约束说明

## Dependencies

{任务编号列表或 "None"}

## Notes

{可选：评论中的技术约束、需求细节、取舍说明}
```

10. 输出简短摘要，包含分析了多少 issue 或直接输入源、创建了多少任务、新分支名，以及结果依赖链。

## Rules

- 先做需求澄清，再写任务文件；不要把输入源原文直接机械搬运成任务。
- 重点保留会影响实现的技术细节、约束和用户已明确提供的信息。
- 任务文件应足够让 `plantask` 直接接手，而不是重新从零理解问题。
- 基础任务号始终按 `tasks/` 和 `tasks/done/` 的最大现有编号顺延，而不是复用 issue 编号。
- 同一个输入源如果拆成多个任务，必须共享同一个新基础任务号并使用字母后缀，不要拆成连续不同整数号。
- 任务排序统一按基础号升序，再按后缀空值、`a`、`b`、`c` 升序；后续 `plantask` 和 `checktask` 也按这套顺序选择“最小”任务。
- `tasks/` 不需要额外索引文件；任务文件名本身就是索引，命名规则同时定义了唯一标识和处理顺序。
- 任务状态和映射信息只写在具体任务文件里。
- 如果任务源自 issue，必须在标题下方第一行写出 issue 编号映射，保证后续能从 task 反查到 issue。
- 不输入参数时，不要扫全量 issue 后批量生成任务；只处理当前 open issues 中编号最小的一条。
- 写入 task 文件前必须先新建并切换到新分支；不要把新 task 文件直接写在原分支上。
- 默认直接产出任务文件，不把“等待用户二次确认”当成标准中间步骤；无法安全假设时才提问。
- 不要提交代码。文件创建就是这个 skill 的最终副作用。
