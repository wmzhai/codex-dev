# TaskDev

`codev-taskdev` 是 task plan 的手动执行器。它把已经审核通过的 task 文件转成实际代码改动，并把实现过程同步回任务文档。

它只覆盖 task 分支内的“选择任务 -> 校准 plan -> 编码 -> 持续同步 task -> 实现收尾精简 -> 默认 build / 最小编译校验”这一段，不负责自动化功能验证、完整 QA、分支部署、归档或主干收尾。

## 第一规则：先用中文交流

- skill 一触发，第一句就用中文和用户交流。
- 整个执行过程中的进度更新、阻塞说明、实现摘要和结果汇报默认都用中文。
- 只有用户明确要求英文或双语时，才切换语言。

## Inputs

从用户请求推断目标任务：

- 如果用户指定了任务编号或文件名，如 `T08` 或 `T08-foo.md`，使用该任务。
- 否则使用 `tasks/` 下任务号最小的待办任务文件，只识别文件名匹配 `^T\\d+-.*\\.md$` 的任务文件，并忽略 `tasks/done/` 与其他非任务文件。

## End State

完成时应尽量达到以下状态：

- 目标 task 已在对应 task 分支上完成当前 plan 范围内的代码实现。
- 对应 `tasks/Txx-*.md` 已同步实际采用的实现路径、重要代码改动、待执行人工验证和剩余缺口。
- 已在收尾阶段自动执行过一次 `codev-simplify` 式语义不变精简；如果没有安全可做的精简，也已明确记录。
- 已在收尾阶段主动执行过一次仓库约定的 build / 最小编译校验；如果入口不可稳定判定或执行失败，应明确停在阻塞态而不是假装可以直接人工验收。
- 当前结果已经适合进入人工验证；人工验证通过后，再进入 `$codev-quickship`。

`codev-taskdev` 默认不负责以下动作：

- 不自动启动本地服务，不自动跑测试、lint、浏览器验证或其他扩展验证动作；但会在收尾阶段主动跑一次仓库默认 build / 最小编译校验
- 不自动做完整 QA 或分支部署
- 不更新 `tasks/` 以外的 `docs/`、`memory/` 或 `AGENTS.md`
- 不归档到 `tasks/done/`
- 不 merge 到 `main/master`
- 不 bump 版本号、打 tag、创建 PR
- 不默认 commit 或 push；需要稳定中间点时，后续单独使用 `$codev-checkpoint`

## Workflow

1. 定位仓库根目录并选择目标任务文件；如果没有待办任务，明确告知并停止。
2. 基于目标任务文件确定工作分支：
   - 取任务文件去掉 `.md` 后的文件名，作为该任务的工作分支名。
3. 检查当前 git 分支与工作区：
   - 如果当前已在目标任务分支，继续后续步骤。
   - 如果当前不在目标任务分支，且工作区足以安全切换，则切到该任务分支。
   - 如果该分支不存在，就基于当前 `HEAD` 新建并切过去。
   - 如果存在未提交改动导致无法安全切换分支，明确报告阻塞并停止。
4. 完整读取任务文件，提取：
   - 目标与背景
   - 验收标准
   - 依赖任务
   - 约束、备注、相关代码入口
   - 已有的 `Implementation Plan` 与 `Validation Plan`
5. 检查依赖任务是否已完成：
   - 如果前置任务未完成或明显缺前提，不要绕过。
   - 先把阻塞原因写入任务文件，再向用户说明并停止。
6. 初始化或补齐任务文件中的过程记录区：
   - 保留原有目标、范围和验收结构。
   - 保留并持续维护已有的 `Implementation Plan` 与 `Validation Plan`。
   - 如有必要，补充 `## Status`、`## Execution Log`、`## Implementation Notes`、`## Pending Validation`、`## Remaining Gaps` 等小节。
7. 在开始编码前先校准 task plan：
   - 优先完整读取任务文件中已有的 `## Implementation Plan` 与 `## Validation Plan`。
   - 如果代码现状与原计划有轻微漂移，先在任务文件里直接更新这些小节，再开始编码。
   - 如果漂移已经上升到产品范围或架构路径冲突，明确说明并停止；必要时建议用户回到 task 文件继续收敛，或补跑 `$plan-eng-review` / `$autoplan`。
8. 分阶段实现代码：
   - 优先按容易收敛的小块推进，而不是一次性铺开。
   - 优先复用当前 repo 已有模式，而不是凭空设计新结构。
   - 每完成一个重要里程碑，就更新一次任务文件中的执行记录和实现说明。
9. 不主动执行除默认 build 之外的其他验证动作：
   - 不自动启动本地服务，不自动跑测试、lint、脚本检查或浏览器验证。
   - 不默认触发 `$design-review`、`$review`、`$qa` 或部署；这些属于后续独立门禁。
   - `Validation Plan` 继续保留给后续人工验证或独立验证门禁使用。
10. 在实现收尾时自动执行一次语义不变精简：
   - 优先沿用 `codev-simplify` 的约束：不改行为、不改公共 API、不引入新依赖、不无故扩大范围。
   - 只在本次 task 相关、已经完成实现的最小必要 diff 上执行这一次；不要在中途重复多轮精简。
   - 如果没有把握，就保持当前实现不改，并在结果里明确说明“没有安全可做的精简”。
11. 精简后主动跑一次仓库默认 build / 最小编译校验：
   - 先根据仓库已有说明和入口定位默认 build 命令，优先读根 `AGENTS.md`、`README.md`、`memory/`、已有脚本或 manifest，不要凭空发明命令。
   - 优先使用仓库已经稳定存在的单一入口，例如 `cargo build --workspace`、`pnpm build`、`npm run build`、`turbo build`、`just build`、`make build` 等同类默认命令。
   - 这一步是 `codev-taskdev` 收尾的默认代码门禁，不替代人工功能验证，也不扩展成完整测试矩阵、浏览器 QA 或部署前验收。
   - 如果 build 失败、命令不可稳定判定、或明显依赖当前不可用环境，要明确报告并停止，不要假装 task 已进入可验收状态。
12. build 后再次同步任务文件：
   - 实际实现与原计划的差异
   - 已改动的重要文件与模块
   - 精简结果或未精简原因
   - 默认 build / 最小编译校验的命令与结果
   - 尚未执行的人工验证、待下游门禁和当前已知缺口
   - 剩余缺口、风险和建议的下一步
   - 当前状态建议写成“实现完成，待人工验证”或语义等价的表述
13. 用中文向用户汇报：
   - 当前 task
   - 当前分支
   - 已完成的代码实现
   - 本次自动精简的结果
   - 默认 build / 最小编译校验的命令与结果
   - 尚未执行的人工验证和下游门禁
   - 下一步建议先人工验证，确认通过后再用 `$codev-quickship`

## Rules

- 默认一次只处理一个 task。
- 默认按最小整数任务号选择待办任务，不扫描 `tasks/done/`。
- 任务文件不是只读输入；实际实现路径变化时，必须按事实同步 task。
- 自动精简是 `codev-taskdev` 的内置收尾步骤，不需要在同一轮实现结束后再额外显式调用一次 `$codev-simplify`。
- 默认 build / 最小编译校验是 `codev-taskdev` 的内置收尾步骤；后续 `codev-quickship` 在存在 task 的场景下不应再重复执行同一门禁，除非用户明确要求复验。
- 不要为了“顺手收口”而替用户执行验证、QA、部署、归档或发布。
- 不要为了默认 build 之外的验证而额外启动本地服务或补搭环境。
- 不要把 task 收尾同步扩大成 repo 级文档巡检；`docs/`、`memory/`、`AGENTS.md` 的更新留给后续 `codev-quickship`。
- 不要假设必须 commit 才算完成；`codev-taskdev` 默认停在代码与任务文档都已更新的实现阶段。

## Hard Blockers

只有在以下情况才停下来问用户：

- 产品范围或行为出现冲突性决策，且无法从 task、repo 或现有模式中安全收敛
- 依赖任务未完成，继续执行会制造错误前提
- 环境或未提交改动导致无法安全切到目标 task 分支
- 实现过程中暴露出多条高影响实现路径，无法安全默认选择

如果遇到硬阻塞：

- 先把当前状态和阻塞原因写入任务文件
- 再用中文向用户说明已经完成到哪一步、缺什么、需要用户给出什么最小输入

## Output

默认返回中文总结，不输出大段源码或完整 patch。

结果中至少包含：

- 任务文件名
- 当前分支
- 当前阶段状态
- 已完成的重要实现
- 本次自动精简的结果
- 默认 build / 最小编译校验的命令与结果
- 尚未执行的人工验证或待下游门禁
- 剩余风险或待验证项
- 下一步推荐的手动动作，默认是“先人工验证，再运行 `codev-quickship`”
