# AutoDev

`autodev` 是 task 阶段的自动执行器。它从 repo 内已有的待办任务出发，在对应 task 分支上推进规划、编码、验证、分支部署和任务文档同步，直到用户可以直接查看已部署结果，并决定是否进入 `automerge`。

它内含 `$taskdev` 的任务选择、plan 校准和编码阶段，因此不需要先显式运行 `$taskdev`。

它把现有 skills 当作工具箱，而不是硬性依赖链路。能直接复用的步骤优先复用；如果现有 skill 不能完整满足当前环节需求，就直接按本 skill 的工作步骤执行，不要为了“复用”而丢失必要行为。

## 第一规则：先用中文交流

- skill 一触发，第一句就用中文和用户交流。
- 整个执行过程中的进度更新、风险说明、阻塞说明、验证结果和收尾总结默认都用中文。
- 只有用户明确要求英文或双语时，才切换语言。

## Inputs

从用户请求推断目标任务：

- 如果用户指定了任务编号或文件名，如 `T08` 或 `T08-foo.md`，使用该任务。
- 否则使用 `tasks/` 下任务号最小的待办任务文件，只识别文件名匹配 `^T\\d+-.*\\.md$` 的任务文件，并忽略 `tasks/done/` 与其他非任务文件。

## End State

完成时必须同时满足以下条件：

- 目标 task 的代码实现已经完成，并在 task 分支上提交、推送。
- 需要的结构审查、UI 审查、测试和部署后验证已经跑完。
- 当前分支已经部署到可供验证的非主干环境，例如 preview、staging 或 branch deploy。
- 对应 `tasks/Txx-*.md` 已按最新实际情况详细更新，足以独立复盘本次任务的实现、验证和部署过程。
- 当前结果已经处于“可供人类确认”的状态。

`autodev` 在这个状态停止，不做以下事情：

- 不 merge 到 `main/master`
- 不 bump 版本号
- 不打 tag
- 不把任务归档到 `tasks/done/`

这些动作留给 `automerge`。

## Workflow

1. 定位仓库根目录并选择目标任务文件；如果没有待办任务，明确告知并停止。
2. 基于目标任务文件确定工作分支：
   - 取任务文件去掉 `.md` 后的文件名，作为该任务的工作分支名。
3. 检查当前 git 分支：
   - 如果当前分支不是 `main` 或 `master`，继续后续步骤。
   - 如果当前分支是 `main` 或 `master`，先切到上一步得到的任务分支。
   - 如果该分支不存在，就基于当前 `HEAD` 新建并切过去。
4. 完整读取任务文件，提取：
   - 目标与背景
   - 验收标准
   - 依赖任务
   - 约束、备注、相关代码入口
   - 已有的 `Implementation Plan` 与 `Validation Plan`
5. 检查依赖任务是否已完成：
   - 如果前置任务未完成或明显缺前提，不要绕过。
   - 先把阻塞原因写入任务文件，再向用户说明并停止。
6. 在开始编码前，先检查当前仓库是否具备安全的分支部署路径：
   - 优先查 repo 文档、部署配置、`CLAUDE.md`、CI 配置和现有脚本。
   - 优先使用 preview、staging、branch deploy 或等价的非主干验证环境。
   - 如果当前仓库只能通过 merge 主干触发部署，不要把 merge 当成部署手段；把这件事记录为阻塞并停止。
7. 初始化或补齐任务文件中的过程记录区：
   - 保留原有目标、范围和验收结构。
   - 保留并持续维护已有的 `Implementation Plan` 与 `Validation Plan`。
   - 如有必要，补充 `## Status`、`## Execution Log`、`## Implementation Notes`、`## Validation Notes`、`## Deployment Notes`、`## Remaining Risks / Pending Human Confirmation` 等小节。
8. 进入实现前先执行内置 `$taskdev` 阶段并校准 task plan：
   - 优先完整读取任务文件中已有的 `## Implementation Plan` 与 `## Validation Plan`。
   - 如果代码现状、依赖状态、部署前提或用户最新确认让原计划不再成立，先在任务文件里直接更新这些小节，再开始编码。
   - 如果当前任务已经有足够清晰的实现方向，也仍要把最终采用的实现路径、关键假设和放弃方案的原因同步回任务文件，再开始编码。
9. 分阶段实现代码：
   - 优先按可验证的小块推进，而不是一次性铺开。
   - 每完成一个重要里程碑，就更新一次任务文件中的执行记录和实现说明。
10. 当 patch 达到一个稳定里程碑时，优先做语义不变精简：
   - 如果 `simplify` 适用，按它的约束收窄 patch。
   - 如果当前改动太早、不适合精简，就继续实现，等到第一个稳定点再做。
11. 在需要 clean working tree 的环节前，先把分支收成稳定状态：
   - 优先 `simplify`
   - 再用普通 commit 或 `checkpoint` 创建中间稳定点并推送当前分支
12. 进入验证循环：
   - 涉及用户可见 UI 时，执行 `design-review` 或等价视觉审查。
   - 对当前 diff 执行结构性 code review；能直接复用 `review` 就复用。
   - 执行当前任务需要的测试与浏览器验证；能直接复用 `qa` 就复用。
   - 如果发现局部可修复问题，直接修复并继续，不要为常规小问题停下来问用户。
13. 每次关键修复后，按最小必要范围补跑验证：
   - 只补跑受影响的测试、审查或验证链，不从头重跑整条流程。
   - 如果自动修复让 patch 明显扩大，可再做一次 `simplify`，随后重新收成 clean tree。
   - 每轮结论都要写回任务文件，包括失败原因、修复思路和最终状态。
14. 把当前 task 分支部署到可验证环境：
   - 优先复用现有部署脚本、平台命令或 CI 工作流。
   - 如果现有 skill 能安全覆盖这一步，就用现有 skill；否则直接按 repo 的部署方式执行。
   - 不要通过 merge `main/master`、不要通过 bump 版本号来换取部署。
15. 做部署后验证：
   - 记录部署 URL、环境名、构建号或等价标识。
   - 运行最小但足够说明结果的 smoke checks、关键路径验证或页面检查。
   - 如果部署后暴露出局部可修复问题，直接回到实现与最小回归链。
16. 收尾前再次完整同步任务文件：
   - 验收标准的当前状态
   - 实际实现与原任务描述的差异
   - 执行过的重要步骤
   - 关键验证结果
   - 部署信息
   - 剩余风险和待人工确认项
   - 当前状态建议写成“已部署待人工确认”或语义等价的表述
17. 用中文向用户汇报：
   - 当前 task
   - 当前分支
   - 已部署结果的访问方式
   - 已完成的验证
   - 仍存风险或确认点
   - 下一步请在确认无误后运行 `automerge`

## Task Document Rules

任务文件是 `autodev` 的过程记录面，不只是最后验收时才回填的 checklist。

- 开始执行前先完整读取一次。
- 执行过程中持续更新，不要只在结束时一次性补写。
- 至少在这些时点更新：
  - 选定实现路径后
  - 完成关键代码里程碑后
  - 完成一次重要审查或测试后
  - 完成部署后
  - 因阻塞停止前
- 必须记录的事实包括：
  - 实际采用的实现方案与关键决策
  - 重要代码改动摘要
  - 验证结果、失败与修复
  - 部署信息与验证入口
  - 剩余风险、限制、待人工确认项
- 允许补充标准小节，但不要破坏原任务文件的主结构和可读性。
- 只写入已经执行、已经验证、可以追溯的事实；不要写猜测、承诺或未来式占位。
- 在 `autodev` 结束时，任务文件必须仍保留在 `tasks/`，不要移动到 `tasks/done/`。

## Reuse Policy

优先把现有 skills 当工具箱使用，但不要机械拼接。

- 直接适用时优先复用：
  - `$taskdev` 的同类约束已经内含在 `autodev` 里，不要再额外显式调用一次
  - `simplify`
  - `checkpoint`
  - `design-review`
  - `review`
  - `qa`
- 不要直接把以下 skill 当作 `autodev` 的收尾步骤：
  - `checktask`：它默认会归档到 `tasks/done/`，不适合 `autodev` 的分支内收尾状态
  - `ship`：它通常伴随正式发布动作，可能引入版本号、CHANGELOG 或 PR 语义
  - `land-and-deploy`：它的默认语义包含合并主干，不适合 `autodev`
- 如果某个现有 skill 只能覆盖一部分需求，就只复用其中合适的部分约束，由 `autodev` 自己接管剩余步骤。

## Hard Blockers

只有在以下情况才停下来问用户：

- 需要人类登录、授权、审批或提供外部凭证
- 产品范围或行为出现冲突性决策，且无法从 repo、任务文档或现有模式中安全收敛
- 测试、QA 或部署后验证暴露出多条高影响修复路径，无法安全默认选择
- 环境限制导致无法在非主干分支上可靠部署或验证
- 前置依赖任务未完成，继续执行会制造错误前提

如果遇到硬阻塞：

- 先把当前状态和阻塞原因写入任务文件
- 再用中文向用户说明已经完成到哪一步、缺什么、需要用户给出什么最小输入

## Output

默认返回中文总结，不输出大段源码或完整 patch。

结果中至少包含：

- 任务文件名
- 当前分支
- 当前阶段状态
- 已完成的重要步骤
- 已部署环境与验证入口
- 剩余风险或待确认项
- 下一步应该运行 `automerge`，还是先等待用户补充阻塞输入
