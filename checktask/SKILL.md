# CheckTask

检查任务文件、更新通过项，并在验收流程末尾最多一次对本次相关 diff 执行 `simplify` 式的语义不变精简；全部通过时再归档任务。

以验收和状态更新为主，不主动修复代码问题；允许在流程末尾附带一次安全、局部的语义不变精简。

## Inputs

从用户请求推断目标任务：

- 如果用户指定了任务编号或文件名，如 `T04`、`T18a` 或 `T18a-foo.md`，使用该任务。
- 否则使用 `tasks/` 下任务顺序最靠前的待办任务文件，只识别文件名匹配 `^T\\d+[a-z]?-.*\\.md$` 的任务文件，并忽略 `tasks/done/` 与其他非任务文件；任务顺序定义为基础号升序，再按后缀空值、`a`、`b`、`c` 升序，因此会先验收 `T18a`，再验收 `T18b`，再验收 `T18c`。

## Workflow

1. 定位仓库根目录并按上述顺序选择目标任务文件。如果没有待办任务，明确告知所有任务已完成并停止。
2. 读取任务文件，提取 `## Acceptance Criteria` 或 `## 验收标准` 下的 checkbox 条目，兼容 bullet 和 numbered checkbox。
3. 逐项验证验收标准，优先使用最小侵入的方法：
   - 代码或配置类标准：读文件、搜索实现、核对结构。
   - 运行时标准：只在标准明确要求时运行针对性的命令或测试。
   - 含糊标准：标记为需要人工确认，不猜测通过。
4. 将明确通过的条目从 `[ ]` 更新为 `[x]`，失败或不确定的条目保持不变。
5. 如果验证过程中发现任务文档与当前实现或实际运行结果不一致，且能确认这是用户在执行过程中做出的实际修改：
   - 以已验证的当前实现和运行结果为准，不以过期文案为准。
   - 同步更新任务文件中相关部分，包括任务描述、范围说明、字段名、约束、示例，以及必要时对应的验收条目表述。
   - 只写入已经验证过的事实，不补写未确认行为，也不要把未完成事项改写成已完成。
6. 如果存在未通过或待人工确认的条目：
   - 保留任务文件在原位置。
   - 向用户报告剩余缺口，不主动修复。
7. 如果所有条目都通过：
   - 再次检查任务文件是否已反映最终验证结果；如果仍有与实现不一致的描述，继续同步，保持文件结构不变。
   - 确保 `tasks/done/` 存在。
   - 在 git 仓库内优先使用 `git mv` 归档到 `tasks/done/`，否则使用普通移动。
8. 在上述验收步骤结束后，最多自动执行一次 `simplify`：
   - 无论是否全部通过，都只在整个 `checktask` 流程末尾执行这一次；不要在中途重复调用。
   - 输入使用本次任务相关、且当前工作区里最小必要范围的 diff；如果用户已明确给出范围，则沿用该范围。
   - 目标是语义不变地降低嵌套、消除重复、缩小 patch；没有把握就不改。
   - 如果不存在可归属到本任务的相关 diff，明确说明“没有可精简的相关改动”，不要编造建议。
9. 返回简洁结果块，包含任务名、每条标准的状态、总通过数、任务是否已归档，以及本次自动精简的简短结果摘要。

## Rules

- 不要提交、推送或发版。
- 不要静默运行大而全的 CI；只运行验证当前标准所需的最小命令。
- 如果用户要求 review 风格输出，先给 findings，再给汇总。
- 如果验证命令因环境限制失败，要明确说明限制，不要把它当作产品缺陷。
- 不要为了“让任务通过”而伪造实现事实或随意放宽要求；只有在验证确认当前实现已因用户实际修改而变化时，才同步 `tasks` 文档中的相关描述和验收条目。
- 文档与实现冲突时，以已验证的实际代码和运行结果为准；过期 task 文案应被修正，而不是反过来压实现。
- 识别任务编号时兼容 `T18a` 这类带字母后缀的子任务。
- `tasks/` 中任务文件名本身就是索引和排序依据。
- 自动执行的精简必须遵守 `simplify` 的约束：不改行为、不改公共 API、不引入新依赖、不无故扩大改动范围。
- 自动精简结果默认只做摘要说明，不回显完整 diff 或大段源码，除非用户明确要求查看。
